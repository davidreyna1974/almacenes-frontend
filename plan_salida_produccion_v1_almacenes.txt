================================================================================
1. RESUMEN EJECUTIVO
================================================================================

El sistema Almacenes es un WMS (Warehouse Management System) de escala media
desarrollado con Angular 21 (frontend) y Spring Boot 3.5 / Java 17 (backend),
persistido en PostgreSQL. Está dirigido a un almacén único con menos de 100
usuarios concurrentes y busca reemplazar hojas de cálculo como herramienta de
gestión operativa.

ESTADO GENERAL: APTO PARA PRODUCCIÓN CON CONDICIONES

El código de negocio, la seguridad de dominio, la cobertura de tests y la
arquitectura general son sólidos y están por encima del estándar mínimo para
sistemas de esta escala. Los problemas identificados no son de lógica de negocio
— son de infraestructura y configuración de despliegue, que es el patrón
habitual de un proyecto que no ha pasado aún por el proceso de "hardening" para
producción.

Antes del go-live DEBEN resolverse los ítems clasificados como BLOQUEANTE (§8).
Los ítems IMPORTANTES deben resolverse en el primer sprint post-go-live (máximo
2 semanas). Los ítems de v1.1+ son mejoras aceptadas como deuda técnica
conocida.

MÉTRICAS ACTUALES DEL SISTEMA:

MÓDULOS IMPLEMENTADOS EN v1.0
------------------------------
  Módulo 0 — Infraestructura base
    Layout shell (sidebar + topbar + main-layout), componentes shared
    (EmptyState, ConfirmDialog, MasterDetail), tema Angular Material
    (paleta #6B3C6B / #CE6EEB / #F2E4F2), interceptores JWT y de errores.

  Módulo 1 — Autenticación
    Login con JWT (HMAC-SHA256), rate limiting (5 intentos → bloqueo 15 min),
    AuthGuard por rol, logout, sidebar dinámico por rol.

  Módulo 2 — Inventario
    Categorías (CRUD), Productos (CRUD + Kardex + movimientos IN/OUT),
    LowStock (alertas visuales por severidad), búsqueda insensible a acentos
    via f_unaccent() de PostgreSQL.

  Módulo 3 — Compras
    Proveedores (CRUD), Órdenes de compra (flujo PENDING→APPROVED→RECEIVED
    o CANCELLED), líneas de detalle con autocomplete de productos.

  Módulo 4 — Ventas
    Clientes (CRUD), Órdenes de venta (flujo PENDING→APPROVED→DELIVERED
    o CANCELLED), reservas de stock con Optimistic Locking, dashboard de
    reservas por producto y por cliente.

  Módulo 5 — Reportes
    V1 Pendientes (OC/OV con navegación a órdenes), V2 Operativos (Stock
    bajo, Kardex, Movimientos, Rotación), V3 Analítico (Rentabilidad,
    Tendencia, Top Productos, ABC, Por Proveedor), V4 Ejecutivo (KPIs +
    gráfica de valuación).

  Módulo 6 — Usuarios
    Gestión de usuarios (CRUD + asignación de roles, solo ADMIN), perfil
    propio (todos los roles), cambio de contraseña, dropdown de usuario
    en topbar.

ROLES DEL SISTEMA
-----------------
  ADMIN          Acceso total — gestión de usuarios + todos los módulos
  MANAGER        Todos los módulos excepto gestión de usuarios + reportes analíticos
  WAREHOUSEMAN   Inventario (lectura), recepción de compras, entrega de ventas,
                 reportes operativos
  SALES          Inventario (lectura), creación y entrega de órdenes de venta,
                 reportes de pendientes

FORTALEZAS CONFIRMADAS POR ANÁLISIS DE CÓDIGO

  ✔ RBAC implementado en doble capa: Spring Security (backend) + authGuard
    (frontend). Tests automatizados de seguridad por rol verifican ambas capas.
  ✔ Optimistic Locking (@Version) en Product para concurrencia de reservas.

  ✔ Documentación técnica exhaustiva: 15+ memorias técnicas, 500+ casos de

4. ANÁLISIS DE BRECHAS — BACKEND
================================================================================

--- BLOQUEANTES ---------------------------------------------------------------

[BACK-B1] Sin herramienta de migraciones de base de datos (Flyway / Liquibase)
  Archivo: pom.xml (ausente), application.yaml (ddl-auto: validate)
  Descripción: No existe ningún script SQL de creación del schema. No hay
  data.sql, schema.sql ni migraciones versionadas. El schema en producción debe
  crearse manualmente. Con ddl-auto: validate, si el schema no existe o está
  desactualizado, la aplicación NO ARRANCA y no hay mensaje que indique qué
  falta exactamente.
  Sin migraciones no es posible:
    · Reproducir la BD en un nuevo servidor sin documentación extra.
    · Rastrear cambios de schema entre versiones.
    · Hacer rollback de un cambio de schema.
  Acción: generar el script DDL inicial desde el ORM (spring.jpa.generate-ddl:
  true una vez para exportar), versionarlo como V1__init_schema.sql y agregar
  la dependencia de Flyway al pom.xml. Para v1.0, un único script SQL es
  aceptable — no es necesaria la complejidad completa de Flyway si el equipo
  prefiere scripts manuales, pero el script DEBE existir en el repositorio.

[BACK-B2] Sin Dockerfile ni docker-compose.yml
  El despliegue es enteramente manual. No hay forma reproducible de levantar
  el stack (app + PostgreSQL + frontend + red) en un servidor nuevo. Cada
  despliegue depende de documentación externa o memoria del operador.
  Acción: crear Dockerfile multi-stage para el backend, Dockerfile multi-stage
  para el frontend (Node.js build → nginx-alpine), y un docker-compose.yml con
  los 3 servicios: db, backend y frontend.
  Ver §9 (Procedimiento de instalación) para los archivos concretos.

[BACK-B3] Sin perfil de producción (application-prod.yaml)
  application.yaml es el único archivo de configuración. Contiene:
    · spring.jpa.show-sql: true  ← imprime cada SQL en producción
    · JWT secret con valor por defecto hardcodeado si JWT_SECRET no está
      definido en el entorno
    · Contraseña de BD por defecto (david123) si DB_PASSWORD no está definido
  Si se olvida definir alguna variable de entorno en producción, la aplicación
  arranca con valores de desarrollo inseguros sin ningún aviso.
  Acción: crear src/main/resources/application-prod.yaml con:
    spring.jpa.show-sql: false
    spring.jpa.properties.hibernate.format_sql: false
    spring.jpa.hibernate.ddl-auto: validate
    springdoc.api-docs.enabled: false
    springdoc.swagger-ui.enabled: false
    logging.level.root: WARN
    logging.level.com.codigo2enter.almacenes: INFO
  Y activar con SPRING_PROFILES_ACTIVE=prod en el servidor de producción.

--- IMPORTANTES ---------------------------------------------------------------

[BACK-I1] DataInitializer registra la contraseña del admin en texto plano
  Archivo: DataInitializer.java, línea 53
  Código: log.warn(" Usuario: admin | Password: Admin123!")
  En los logs de arranque del servidor queda impresa la contraseña en claro.
  Cualquier persona con acceso a los logs del servidor la ve inmediatamente.
  Acción: eliminar esa línea. El log útil es solo confirmar que el admin fue
  creado. La contraseña NUNCA debe loggearse.

[BACK-I2] Swagger UI accesible sin autenticación en producción
  Archivo: SecurityConfig.java (permitAll en /swagger-ui/**, /v3/api-docs/**)
  En producción, cualquier persona que llegue al servidor puede explorar toda
  la estructura de la API (endpoints, DTOs, parámetros, respuestas posibles)
  sin autenticarse. Esto facilita la enumeración de la API por atacantes.
  Acción: en application-prod.yaml: springdoc.api-docs.enabled: false
  O restringir la ruta a rangos IP de red interna en SecurityConfig.

[BACK-I3] Sin headers de seguridad HTTP
  Archivo: SecurityConfig.java
  El servidor no envía ninguno de los headers de seguridad estándar:
    · Content-Security-Policy
    · Strict-Transport-Security (HSTS)
    · X-Frame-Options: DENY
    · X-Content-Type-Options: nosniff
    · Referrer-Policy: strict-origin-when-cross-origin
  Acción en SecurityConfig.java (método filterChain):
    http.headers(headers -> headers
        .frameOptions(f -> f.deny())
        .xssProtection(Customizer.withDefaults())
        .contentTypeOptions(Customizer.withDefaults())
        .httpStrictTransportSecurity(hsts -> hsts
            .includeSubDomains(true)
            .maxAgeInSeconds(31536000))
    );
  El CSP completo configurarlo en nginx (más flexible que en Spring).

[BACK-I4] DataIntegrityViolationException no manejada en GlobalExceptionHandler
  Archivo: GlobalExceptionHandler.java
  Si una constraint de BD se viola por race condition (dos requests simultáneos
  crean el mismo username/RFC), PostgreSQL lanza DataIntegrityViolationException
  que llega al catch genérico de RuntimeException y retorna HTTP 500 exponiendo
  el nombre interno de la constraint (ej: "uq_supplier_rfc").
  Acción: agregar handler específico:
    @ExceptionHandler(DataIntegrityViolationException.class)
    public ResponseEntity<ApiError> handleDataIntegrity(...) {
        // Extraer constraint del mensaje para dar mensaje amigable
        // Retornar 409 con mensaje genérico
    }

[BACK-I5] ~15 new RuntimeException() en servicios que deberían ser
  ResourceNotFoundException
  Archivos: PurchaseOrderServiceImpl, SaleOrderServiceImpl,
  ReservationServiceImpl (~15 ocurrencias total)
  Ejemplo: "Orden con id X no encontrada" lanzada como RuntimeException →
  GlobalExceptionHandler lo captura como HTTP 500, no como 404.
  Esto significa que un GET a una orden inexistente devuelve 500 en lugar de 404,
  contamina los logs de producción (los errores 500 parecen bugs del sistema)
  y dificulta el monitoreo.
  Acción: buscar con grep "new RuntimeException" en los servicios y reemplazar
  por ResourceNotFoundException o BusinessRuleException según corresponda.

[BACK-I6] Login devuelve HTTP 500 para credenciales incorrectas (no 401)
  El backend lanza BadCredentialsException que sí es capturada por
  GlobalExceptionHandler y retorna 401, PERO si hay algún otro error en el
  flujo de autenticación, puede retornar 500. El frontend ya lo maneja leyendo
  err.error?.message independientemente del status code. Sin embargo, en
  producción, cada login fallido por contraseña incorrecta genera un 500 en
  los logs, lo que hace imposible distinguir errores reales del servidor de
  intentos de login fallidos con alertas basadas en tasa de 5xx.
  Acción: verificar que AuthController siempre retorna 401 para credenciales
  incorrectas (nunca 500). Si hay un caso que llega a 500, corregirlo.

[BACK-I7] Sin configuración de HikariCP
  application.yaml no tiene spring.datasource.hikari.*. Se usan los defaults:
  pool de 10 conexiones, timeout 30 segundos. Para producción con carga normal
  de un almacén, 10 conexiones puede ser insuficiente durante picos (generación
  de reportes analíticos que ejecutan múltiples queries simultáneas).
  Acción en application-prod.yaml:
    spring.datasource.hikari.maximum-pool-size: 20
    spring.datasource.hikari.minimum-idle: 5
    spring.datasource.hikari.connection-timeout: 30000
    spring.datasource.hikari.idle-timeout: 600000
    spring.datasource.hikari.max-lifetime: 1800000

[BACK-I8] Sin Spring Boot Actuator
  pom.xml no incluye spring-boot-starter-actuator. Sin él no hay:
    · /actuator/health para verificar que el sistema está vivo
    · Liveness/readiness probes para load balancers
    · Métricas de JVM y pool de conexiones
    · Smoke tests automatizados post-deploy
  Acción: agregar al pom.xml y configurar en application-prod.yaml:
    management.endpoints.web.exposure.include: health,info
    management.endpoint.health.show-details: never
    management.server.port: 8081  # Puerto separado, no expuesto al público

[BACK-I9] Sin logback-spring.xml (logging estructurado para producción)
  Spring Boot usa su configuración por defecto: solo consola, nivel INFO, sin
  rotación, sin formato JSON. En producción esto significa:
    · Los logs se pierden al reiniciar el proceso
    · No hay integración posible con sistemas de monitoreo (ELK, Grafana Loki)
    · No hay rotación automática (el log crece indefinidamente en disco)
  Acción: crear src/main/resources/logback-spring.xml con:
    · Perfil dev: consola con colores
    · Perfil prod: archivo rotativo diario + retención 30 días + nivel WARN
      para librerías externas, INFO para com.codigo2enter.almacenes

[BACK-I10] Sin logging de operaciones críticas de negocio
  Ningún servicio tiene @Slf4j + registro de las operaciones de negocio
  importantes. En producción no hay trazabilidad de:
    · Login exitoso/fallido con username e IP de origen
    · Cambios de contraseña
    · Creación/desactivación de usuarios
    · Transiciones de estado de órdenes (quién aprobó OC-123 y cuándo)
    · Movimientos de stock (quién registró qué movimiento)
  Los campos de auditoría en la BD (createdByUsername, approvedByUsername, etc.)
  cubren la trazabilidad de datos, pero no hay registro en logs de los intentos
  fallidos ni de las acciones exitosas en tiempo real.
  Acción: agregar @Slf4j en los ServiceImpl críticos y log.info() en las
  operaciones de negocio con username, id de entidad y acción.

--- MEJORAS (v1.1) ------------------------------------------------------------

[BACK-M1] User.roles con FetchType.EAGER — JOIN redundante por request
  User.java línea ~50: @ManyToMany(fetch = FetchType.EAGER)
  Los roles ya viajan en el JWT. Cargarlos desde BD en cada request autenticado
  (via resolveAuthenticatedUser) genera un JOIN innecesario. Cambiar a LAZY y

[BACK-M2] Sin @Operation/@ApiResponse en endpoints individuales
  Solo hay @Tag a nivel de clase. Swagger genera documentación básica pero
  sin descripciones de respuestas posibles. Para v1.1, agregar @Operation con
  descripción y @ApiResponse para 200/400/404/422/429.

[BACK-M3] Endpoints auxiliares sin paginación — sin impacto real en v1.0
  VERIFICADO EN CÓDIGO (2026-06-18): el finding original era parcialmente
  incorrecto. Situación real:

  · Endpoint principal de lista (/status/{status}) → SÍ paginado ✓
    PurchaseOrderController línea 98: ResponseEntity<PageResponseDTO<...>>
    SaleOrderController línea 34:     ResponseEntity<PageResponseDTO<...>>
    Ambos aceptan ?page=0&size=20. Es el endpoint que alimenta las tabs
    de la lista principal en el frontend.

  · Endpoints auxiliares por filtro → NO paginados (retornan List<>)

[BACK-M4] Sin @Cacheable en catálogos de lectura frecuente
  Categorías, estados y roles no cambian con frecuencia pero se consultan en

[BACK-M5] ExecutiveReportServiceImpl: fecha por defecto "desde el año 2000"
  Si from es null, se usa LocalDateTime.of(2000, 1, 1, 0, 0). Con datos de
  producción desde 2026, esto es inofensivo a corto plazo pero es un bug latente.
  Mejor default: inicio del año en curso (LocalDate.now().withDayOfYear(1)).

--- BLOQUEANTES ---------------------------------------------------------------

[FRONT-B1] environment.prod.ts tiene URL de API placeholder (dominio ficticio)
  Archivo: src/environments/environment.prod.ts
  Contenido actual: apiUrl: 'https://api.tudominio.com/api/v1'
  La URL queda compilada en el bundle de Angular. Si se hace ng build sin
  reemplazar esta URL, TODAS las peticiones HTTP fallarán en producción.
  Angular no soporta configuración de URL en runtime sin un patrón específico.
  Acción (opción 1 — simple, arquitectura Docker del §9): con nginx proxeando
  /api/ al contenedor backend, el frontend y la API comparten el mismo dominio.
  La URL puede ser una ruta relativa o la misma URL del frontend:
    apiUrl: 'https://almacenes.miempresa.com/api/v1'
  Ventaja adicional: al ser mismo origen, CORS no aplica (el browser no bloquea).
  Acción (opción 2 — para múltiples entornos): implementar un APP_INITIALIZER
  que cargue la config desde src/assets/config.json en runtime:
    { "apiUrl": "https://almacenes.miempresa.com/api/v1" }
  Este archivo no se commitea. El servidor lo sirve con los valores correctos.

[FRONT-B2] angular.json — configuración de producción sin optimización explícita
  Archivo: angular.json, bloque "production"
  La configuración de producción no declara explícitamente:
    "optimization": true,
    "sourceMap": false,
    "namedChunks": false,
    "buildOptimizer": true
  En Angular 21 los defaults del builder aplican optimización automáticamente,
  pero confiar en defaults implícitos es frágil. Más importante: sourceMap sin
  valor explícito false en producción deja la puerta abierta a exponer el código
  fuente accidentalmente.
  Acción: agregar explícitamente en el bloque "production" de angular.json:
    "optimization": true,
    "sourceMap": false,
    "namedChunks": false

[FRONT-B3] Rutas de Inventory sin canActivate ni data.roles
  Archivo: src/app/app.routes.ts (ruta 'inventory')
           src/app/modules/inventory/inventory.routes.ts
  La ruta 'inventory' no tiene canActivate propio ni data.roles. Las rutas
  hijas (products, categories, low-stock) tampoco. El guard del padre ''
  (MainLayout) protege contra acceso sin token, pero el contrato RBAC explícito
  no está declarado. Si en el futuro se agrega una sub-ruta de Inventory
  restringida a ADMIN/MANAGER, heredará el patrón permisivo y quedará
  desprotegida.
  Para v1.0, el acceso de lectura a Inventory para todos los roles es correcto,
  pero la inconsistencia con el resto del proyecto (purchases, sales, reports
  sí tienen canActivate + data.roles) es un riesgo arquitectónico.
  Acción: agregar a la ruta 'inventory' en app.routes.ts:
    canActivate: [authGuard],
    data: { roles: ['ROLE_ADMIN', 'ROLE_MANAGER', 'ROLE_WAREHOUSEMAN', 'ROLE_SALES'] }

--- IMPORTANTES ---------------------------------------------------------------

[FRONT-I1] Suscripciones sin takeUntilDestroyed en componentes de Reports
  Archivos: executive-dashboard.component.ts, analytics-dashboard.component.ts,
            operational-report.component.ts, pending-report.component.ts
  Los métodos load(), loadKardex(), loadMovements(), loadTurnover(), etc. usan
  finalize().subscribe() sin takeUntilDestroyed. Si el usuario navega fuera del
  componente mientras una petición está en vuelo (común en reportes con filtros
  de fecha que tardan segundos), la suscripción no se cancela y puede intentar
  actualizar estado de un componente ya destruido.
  Acción: inyectar DestroyRef en los 4 componentes de Reports y agregar
  .pipe(takeUntilDestroyed(this.destroyRef)) antes de cada .subscribe().

[FRONT-I2] Sin ErrorHandler global de Angular
  Archivo: src/app/app.config.ts
  Angular tiene un sistema de error handling de componentes separado de los
  errores HTTP. Sin un ErrorHandler personalizado, los errores de template
  (null references, errores en ngOnInit), los errores de carga de módulos lazy
  y los RxJS errors no capturados muestran un stack trace en la consola pero no
  hay forma de mostrar al usuario una pantalla amigable ni de registrarlos.
  En producción, el usuario simplemente ve la pantalla congelada.
  Acción: implementar GlobalErrorHandler extends ErrorHandler que:
    · Loggea el error internamente
    · Muestra un snackbar rojo "Ocurrió un error inesperado. Recarga la página."
    · (Opcional v1.1) envía el error a un sistema de monitoreo
  Registrar en app.config.ts: { provide: ErrorHandler, useClass: GlobalErrorHandler }

[FRONT-I3] Sin página 404 personalizada
  Archivo: src/app/app.routes.ts
  El wildcard '**' hace redirectTo: '' en lugar de mostrar un componente de
  error 404. El usuario que navega a una URL incorrecta llega al root sin saber
  por qué fue redirigido.
  Acción: crear NotFoundComponent con mensaje "Página no encontrada" y link al
  inicio. Usar { path: '**', component: NotFoundComponent }.

[FRONT-I4] Sin página de acceso denegado (403)
  authGuard redirige a '/' cuando el rol no tiene acceso. El usuario no recibe
  feedback de por qué fue redirigido.
  Acción: redirigir a '/access-denied' con un componente que explique "No tienes
  permiso para acceder a esta sección" y muestre el rol actual del usuario.

[FRONT-I5] Clase CSS snack-error ausente en global.scss
  Archivos: executive-dashboard, analytics-dashboard, operational-report,
            pending-report — usan panelClass: ['snack-error']

IMPLEMENTADO ✔

[OPS-B1] Sin procedimiento de backup — CRÍTICO para go-live
  No existe ningún documento de procedimiento de backup en el repositorio.
  Si el servidor de BD falla sin backup, se pierde el historial completo de
  órdenes, Kardex, clientes, proveedores y productos.

[OPS-B2] Sin procedimiento de recuperación ante desastres — CRÍTICO para go-live
  No hay documentación sobre qué hacer si el servidor cae o si la BD se
  corrompe. No hay scripts de restauración, no hay lista de variables de
  entorno necesarias para reproducir el entorno.
  Ver §10 para el procedimiento completo.

[OPS-B3] Sin procedimiento de carga inicial de datos — CRÍTICO para go-live
  El sistema sale a producción vacío. Si el cliente tiene datos existentes en
  hojas de cálculo (productos, proveedores, clientes, saldos iniciales de
  stock), la migración inicial debe planificarse ANTES del go-live.
  El sistema solo permite crear registros uno por uno desde la UI.
  Opciones según volumen:
    · < 50 registros: carga manual desde la UI por un operador
    · 50-500 registros: script de importación via API REST (curl/Python)
    · > 500 registros: evaluar una importación directa a PostgreSQL vía
      archivo SQL o pg_restore, verificando las constraints de la BD

[OPS-B4] Sin manual de usuario — IMPORTANTE para go-live
  No existe ningún documento orientado al usuario final. Todo el material del
  repositorio es técnico. Sin manual, el go-live requiere entrenamiento
  extenso en persona.
  Acción mínima para v1.0: crear un documento "Guía rápida por rol" con
  los flujos más comunes para cada uno de los 4 roles.

[OPS-B5] Contraseñas QA documentadas en el repositorio
  memoria_tecnica_global_proyecto.md §5 documenta:
    qa_manager / QaManager123!
    qa_sales / QaSales123!
    qa_warehouse / QaWarehouse123!
    admin / Admin123!
  Estas contraseñas están en el historial de git. En producción DEBEN
  cambiarse. Los usuarios QA no deben existir en el entorno productivo.
  Acción: antes del go-live, cambiar la contraseña de admin desde la UI y
  desactivar todos los usuarios qa_*. Documentar la contraseña del admin de
  producción en un gestor de contraseñas, NUNCA en el repositorio.

[OPS-B6] Sin tests E2E automatizados para el flujo crítico
  Toda la verificación de los flujos de negocio (crear orden → aprobar →
  recibir/entregar → verificar stock) es manual. Cada despliegue requiere
  una regresión manual que, con 500+ casos de prueba documentados, tomaría
  horas. Sin automatización E2E, los deploys son lentos y arriesgados.
  Acción mínima para v1.0: implementar 5-8 specs Cypress o Playwright:
    1. Login con cada rol → verificar sidebar correcto
    2. Crear producto → verificar que aparece en lista
    3. Crear OC → aprobar → recibir → verificar que stock aumentó
    4. Crear OV → aprobar → entregar → verificar que stock decrementó
    5. Acceso por URL a ruta no autorizada → verificar redirección
    6. Crear orden con stock insuficiente → verificar error

[OPS-B7] Sin smoke tests post-deploy
  No hay script de verificación rápida para confirmar que el sistema está
  vivo después de un despliegue. Ver §11 (Checklist de go-live) para el

9. PROCEDIMIENTO DE INSTALACIÓN EN PRODUCCIÓN
================================================================================

REQUISITOS DEL SERVIDOR
-----------------------
  Sistema operativo: Ubuntu 22.04 LTS o RHEL 8/9 (recomendado)
  CPU:               mínimo 2 núcleos (4 recomendado para reportes analíticos)
  RAM:               mínimo 4 GB (2 GB para app + 1.5 GB para PostgreSQL + SO)
  Disco:             mínimo 50 GB SSD (20 GB app + logs + 30 GB BD + crecimiento)
  Red:               1 IP pública fija + dominio con certificado SSL/TLS
  Software base:
    · Docker 24+ (motor de contenedores)
    · Docker Compose v2 (en Linux: sudo apt-get install docker-compose-v2)
    · certbot (solo para obtener el certificado SSL inicial — sin nginx en host)
  Nota: Java 17, PostgreSQL 16, Node.js 20 y nginx corren DENTRO de los
        contenedores Docker. No se instalan directamente en el host.

PASO 1 — OBTENER CERTIFICADO SSL
----------------------------------
  Antes de levantar Docker, obtener el certificado Let's Encrypt con certbot
  en modo standalone. El puerto 80 debe estar libre en el host.

  1.1 Instalar certbot:
      sudo apt-get install -y certbot

  1.2 Obtener el certificado SSL:
      sudo certbot certonly --standalone -d almacenes.miempresa.com
      # Certificados en: /etc/letsencrypt/live/almacenes.miempresa.com/
      #   fullchain.pem  → certificado público
      #   privkey.pem    → clave privada

  1.3 Configurar renovación automática (cron detiene y reinicia el contenedor):
      sudo crontab -e
      # Agregar:
      0 3 * * * certbot renew --quiet \
          --pre-hook  "docker compose -f /opt/almacenes/docker-compose.yml stop frontend" \
          --post-hook "docker compose -f /opt/almacenes/docker-compose.yml start frontend"

PASO 2 — PREPARAR EL SERVIDOR
--------------------------------
  2.1 Instalar Docker y Docker Compose:
      sudo apt-get update
      sudo apt-get install -y docker.io docker-compose-v2
      sudo usermod -aG docker $USER
      newgrp docker   # aplica el grupo sin cerrar sesión

  2.2 Crear directorio de trabajo:
      sudo mkdir -p /opt/almacenes
      sudo chown $USER:$USER /opt/almacenes

PASO 3 — CONFIGURAR VARIABLES DE ENTORNO
-----------------------------------------
  Crear /opt/almacenes/.env (docker-compose lo lee automáticamente):

      # Base de datos
      DB_PASSWORD=contraseña_segura_de_produccion

      # JWT (generar con: openssl rand -hex 64)
      JWT_SECRET=clave_hex_de_64_bytes_minimo_generada_con_openssl

      # CORS — dominio del frontend (sin barra final)
      CORS_ALLOWED_ORIGINS=https://almacenes.miempresa.com

  Asegurar permisos restrictivos:
      chmod 600 /opt/almacenes/.env

  IMPORTANTE: NUNCA commitear .env al repositorio. Agregar .env al .gitignore.
  IMPORTANTE: JWT_SECRET debe tener al menos 64 caracteres hexadecimales.
              Generar con: openssl rand -hex 64

PASO 4 — DESPLEGAR EL STACK DOCKER
-------------------------------------
  4.1 Copiar los archivos al servidor (o clonar los repositorios directamente):
      # Estructura esperada en /opt/almacenes/:
      #   docker-compose.yml
      #   backend/   (código fuente del backend con su Dockerfile)
      #   frontend/  (código fuente del frontend con Dockerfile y nginx.conf)
      scp docker-compose.yml usuario@servidor:/opt/almacenes/
      # Alternativa más limpia: git clone de ambos repos en /opt/almacenes/.

  4.2 IMPORTANTE: actualizar environment.prod.ts con la URL real del API antes
      de construir la imagen. Con nginx en el mismo contenedor proxeando /api/,
      frontend y backend comparten dominio (mismo origen — CORS no aplica):
      # src/environments/environment.prod.ts
      # apiUrl: 'https://almacenes.miempresa.com/api/v1'

  4.3 Construir las imágenes Docker (backend + frontend):
      cd /opt/almacenes
      docker compose build
      # Backend:  Maven compila el JAR → JRE-alpine corre la app
      # Frontend: Node.js compila Angular → nginx-alpine sirve los archivos
      # Tiempo estimado: 3-5 min en primer build; ~1 min en rebuilds
      #                  (capas de npm/maven se cachean automáticamente)

  4.4 Levantar el stack completo:
      docker compose up -d
      # Orden garantizado por depends_on: db → backend → frontend

  4.5 Verificar que los 3 servicios están corriendo:
      docker compose ps
      # Los 3 deben mostrar Status: running (healthy)

  4.6 Verificar logs del arranque del backend:
      docker compose logs -f backend
      # Esperar: "Started AlmacenesApplication in X seconds"
      # Esperar: "Tomcat started on port 8080"
      # No debe aparecer: "Error creating bean" ni "Connection refused"

PASO 5 — INICIALIZAR LA BASE DE DATOS
----------------------------------------
  5.1 Instalar extensión unaccent (OBLIGATORIO para búsquedas con acentos):
      docker compose exec db psql -U almacen_user -d almacen_db \
          -c "CREATE EXTENSION IF NOT EXISTS unaccent;"

  5.2 Crear función f_unaccent (OBLIGATORIO — usada por los índices):
      docker compose exec db psql -U almacen_user -d almacen_db -c "
        CREATE OR REPLACE FUNCTION f_unaccent(text)
        RETURNS text LANGUAGE sql IMMUTABLE PARALLEL SAFE STRICT AS
        \$func\$ SELECT public.unaccent('public.unaccent', \$1) \$func\$;"

  5.3 Crear el schema de la BD (HASTA QUE SE IMPLEMENTE FLYWAY — BACK-B1):
      · Opción A: levantar el backend con ddl-auto: create (solo primer arranque
        con BD vacía). ADVERTENCIA: DESTRUYE los datos existentes.
      · Opción B (recomendada): generar el DDL con Maven en máquina de desarrollo,
        copiar schema.sql al servidor y ejecutar:
          docker compose exec -T db psql -U almacen_user -d almacen_db < schema.sql
        Luego cambiar ddl-auto a validate para todos los despliegues siguientes.

  5.4 Crear índices funcionales para búsqueda insensible a acentos:
      docker compose exec db psql -U almacen_user -d almacen_db << 'SQL'
      CREATE INDEX IF NOT EXISTS idx_products_unaccent_name
          ON products (f_unaccent(lower(name)));
      CREATE INDEX IF NOT EXISTS idx_products_unaccent_sku
          ON products (f_unaccent(lower(sku)));
      CREATE INDEX IF NOT EXISTS idx_categories_unaccent_name
          ON categories (f_unaccent(lower(name)));
      CREATE INDEX IF NOT EXISTS idx_suppliers_unaccent_company
          ON suppliers (f_unaccent(lower(company_name)));
      CREATE INDEX IF NOT EXISTS idx_clients_unaccent_name
          ON clients (f_unaccent(lower(name)));
      CREATE INDEX IF NOT EXISTS idx_users_username ON users (username);
      CREATE INDEX IF NOT EXISTS idx_users_email ON users (email);
      CREATE INDEX IF NOT EXISTS idx_sale_orders_status ON sale_orders (status);
      CREATE INDEX IF NOT EXISTS idx_purchase_orders_status ON purchase_orders (status);
      CREATE INDEX IF NOT EXISTS idx_stock_movements_product_id
          ON stock_movements (product_id);
      SQL

10. PROCEDIMIENTO DE BACKUP Y RECUPERACIÓN ANTE DESASTRES
================================================================================

POLÍTICA DE BACKUP
------------------
  · Tipo:       Lógico (pg_dump — SQL portable)
  · Frecuencia: Diario, a las 02:00 AM (horario de menor actividad)
  · Retención:  Últimos 30 días diarios + 1 backup mensual por 12 meses
  · Verificación: Restaurar en entorno de prueba el primer lunes de cada mes
  · Almacenamiento: Local (/var/backups/almacenes/) + copia remota (S3 o SFTP)

SCRIPT DE BACKUP (/opt/almacenes/scripts/backup.sh)
----------------------------------------------------
  #!/bin/bash
  set -euo pipefail

  BACKUP_DIR="/var/backups/almacenes"
  TIMESTAMP=$(date +%Y%m%d_%H%M%S)
  FILENAME="almacen_db_${TIMESTAMP}.sql.gz"
  RETENTION_DAYS=30

  # Crear directorio si no existe
  mkdir -p "${BACKUP_DIR}"

  # Generar backup comprimido
  PGPASSWORD="${DB_PASSWORD}" pg_dump \
      -h localhost -U almacen_user -d almacen_db \
      --no-owner --no-acl \
      | gzip > "${BACKUP_DIR}/${FILENAME}"

  # Verificar que el archivo no está vacío
  if [ ! -s "${BACKUP_DIR}/${FILENAME}" ]; then
      echo "ERROR: El backup está vacío. Verificar conexión a BD."
      exit 1
  fi

  echo "Backup completado: ${BACKUP_DIR}/${FILENAME} ($(du -h "${BACKUP_DIR}/${FILENAME}" | cut -f1))"

  # Eliminar backups más antiguos que la retención configurada
  find "${BACKUP_DIR}" -name "almacen_db_*.sql.gz" -mtime +${RETENTION_DAYS} -delete

  # (Opcional) Copiar a almacenamiento remoto
  # aws s3 cp "${BACKUP_DIR}/${FILENAME}" s3://bucket-backups/almacenes/

CONFIGURAR CRON (/etc/cron.d/almacenes-backup)
----------------------------------------------
  # Backup diario a las 02:00 AM
  0 2 * * * root /bin/bash /opt/almacenes/scripts/backup.sh >> /var/log/almacenes-backup.log 2>&1

PROCEDIMIENTO DE RESTAURACIÓN
------------------------------
  1. Identificar el backup a restaurar:
     ls -lh /var/backups/almacenes/

  2. Detener el backend para evitar escrituras durante la restauración:
     sudo systemctl stop almacenes-backend

  3. Crear una BD vacía nueva (o limpiar la existente):
     sudo -u postgres psql -c "DROP DATABASE IF EXISTS almacen_db_restore;"
     sudo -u postgres psql -c "CREATE DATABASE almacen_db_restore OWNER almacen_user;"
     sudo -u postgres psql -c "\c almacen_db_restore; CREATE EXTENSION unaccent;"
     sudo -u postgres psql -c "\c almacen_db_restore; CREATE OR REPLACE FUNCTION f_unaccent..."

  4. Restaurar el backup:
     gunzip -c /var/backups/almacenes/almacen_db_20260617_020000.sql.gz \
         | PGPASSWORD="${DB_PASSWORD}" psql -h localhost -U almacen_user almacen_db_restore

  5. Verificar la integridad básica:
     PGPASSWORD="${DB_PASSWORD}" psql -h localhost -U almacen_user almacen_db_restore \
         -c "SELECT COUNT(*) FROM products; SELECT COUNT(*) FROM sale_orders;"

  6. Si la restauración es correcta, hacer swap de la BD activa:
     sudo -u postgres psql -c "ALTER DATABASE almacen_db RENAME TO almacen_db_old;"
     sudo -u postgres psql -c "ALTER DATABASE almacen_db_restore RENAME TO almacen_db;"

  7. Reiniciar el backend:
     sudo systemctl start almacenes-backend

  8. Ejecutar el smoke test (ver §11) para verificar el sistema.

  9. Una vez verificado, eliminar la BD vieja:
     sudo -u postgres psql -c "DROP DATABASE almacen_db_old;"

TIEMPO OBJETIVO DE RECUPERACIÓN (RTO)
--------------------------------------
  · RPO (Punto de recuperación): máximo 24 horas (con backup diario)
  · RTO (Tiempo de recuperación): máximo 2 horas para restauración completa
  · Para reducir RPO a horas: implementar backup cada 4 horas (cambiar cron)
  · Para reducir RPO a minutos: implementar replicación de streaming de PostgreSQL
    (requiere servidor secundario — fuera del alcance de v1.0)

12. ALCANCE FUNCIONAL DECLARADO v1.0 Y RESTRICCIONES CONOCIDAS
================================================================================

ALCANCE v1.0
  · Un solo almacén (no multi-warehouse)
  · Sin gestión de ubicaciones físicas (sin zonas/pasillos/estantes)
  · Sin trazabilidad de lotes ni números de serie
  · Sin estrategia de picking automatizada (FIFO/FEFO)
  · Sin módulo de devoluciones (workaround: movimiento manual de stock)
  · Sin alertas automáticas push (workaround: revisar reporte de stock bajo
    manualmente al inicio del turno)
  · Sin inventario físico asistido por sistema (workaround: usar movimientos
    de ajuste con motivo "AJUSTE INVENTARIO FÍSICO YYYY-MM-DD")
  · Sin importación masiva de datos desde CSV/Excel (carga manual o via API)
  · JWT almacenado en localStorage (no HttpOnly cookie) — riesgo XSS mitigado
    con CSP y dado que la app no renderiza contenido HTML externo no controlado
  · Sin manejo de sesiones concurrentes — JWT válido por 2 horas sin posibilidad
    de invalidación anticipada (consecuencia de arquitectura stateless)

PROCEDIMIENTO OPERATIVO WORKAROUNDS v1.0
  · Devolución de cliente: registrar un movimiento IN con motivo
    "DEVOLUCION CLIENTE - OV-NNN - MOTIVO"
  · Devolución a proveedor: registrar un movimiento OUT con motivo
    "DEVOLUCION PROVEEDOR - OC-NNN - MOTIVO"
  · Inventario físico anual: comparar Kardex del sistema con conteo físico,
    registrar movimiento de ajuste (IN si el sistema tiene menos, OUT si tiene
    más) con motivo "AJUSTE INVENTARIO FÍSICO YYYY-MM-DD"
  · Alertas de stock bajo: WAREHOUSEMAN debe revisar Reportes → Operativos →
    Stock Bajo cada mañana al inicio de turno

13. DEUDA TÉCNICA DOCUMENTADA PARA v1.1+
================================================================================

Esta sección documenta las decisiones de diseño y funcionalidades que se han
evaluado conscientemente y se difieren para versiones posteriores. No son bugs
ni omisiones — son decisiones de alcance.

MÓDULOS FUNCIONALES EN ROADMAP
  · Módulo Returns: devoluciones de clientes y a proveedores. Prioridad alta
    si el cliente tiene volumen de devoluciones > 5% de transacciones.
  · Módulo Notifications: alertas automáticas por email/websocket para stock
    bajo y órdenes vencidas.
  · Módulo Locations: gestión de ubicaciones físicas (zonas, estantes, bins).
    Prerrequisito para FIFO/FEFO automatizado.
  · Módulo Shipping: soporte para despachos con número de guía/transportista.

DECISIONES TÉCNICAS ACEPTADAS
  · Sin Flyway para v1.0: el schema se mantiene con script SQL manual.
    Prioridad alta para v1.1 cuando aparezca el primer cambio de schema.
  · JWT en localStorage: alternativa es HttpOnly cookie + endpoint de refresh,
    requiere cambios en frontend Y backend. Evaluar en v1.1 según el nivel
    de exposición de la aplicación a internet.
  · Sin tests E2E automatizados: la cobertura unitaria es excelente (89.89%
    statements). Los tests E2E manuales están documentados por protocolo
    Propuesta A-D. Automatizar con Cypress en v1.1.
  · Sin Rate Limiting global (solo en login): agregar en v1.1 si hay exposición
    a internet con Spring Security Rate Limiter o API Gateway.
  · Tabla de auditoría general ausente: la trazabilidad actual por campos en
    las entidades es suficiente para v1.0. Una tabla audit_log completa
    (toda entidad, todos los campos) es v1.1+.

LECCIONES TRANSVERSALES DOCUMENTADAS (L1-L35)
  El archivo CLAUDE.md y memoria_tecnica_global_proyecto.md documentan 35
  lecciones aprendidas durante el desarrollo, incluyendo patrones de bugs
  recurrentes y sus soluciones. Estas lecciones deben consultarse antes de
  iniciar cualquier módulo nuevo en v1.1.
