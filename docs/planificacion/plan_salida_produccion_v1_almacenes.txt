1. RESUMEN EJECUTIVO
================================================================================

El sistema Almacenes es un WMS (Warehouse Management System) de escala media
desarrollado para codigoCodigoEnter con Angular 21 (frontend) y Spring Boot
3.5 / Java 17 (backend), persistido en PostgreSQL. Está dirigido a un almacén
único con menos de 100 usuarios concurrentes y busca reemplazar hojas de
cálculo como herramienta de gestión operativa.

ESTADO GENERAL: APTO PARA PRODUCCIÓN CON CONDICIONES

El código de negocio, la seguridad de dominio, la cobertura de tests y la
arquitectura general son sólidos y están por encima del estándar mínimo para
sistemas de esta escala. Los problemas identificados no son de lógica de negocio
— son de infraestructura y configuración de despliegue, que es el patrón
habitual de un proyecto que no ha pasado aún por el proceso de "hardening" para
producción.

Antes del go-live DEBEN resolverse TODOS los ítems clasificados como BLOQUEANTE
(§8): los originalmente bloqueantes, los originalmente importantes, y las mejoras
BACK-M1, BACK-M2, BACK-M5, FRONT-M1–FRONT-M5. Los ítems de v1.1+ (BACK-M3,
BACK-M4 y funcionalidades WMS no críticas) son deuda técnica conocida. Las
brechas WMS "Resolver según alcance con el cliente" (ubicaciones, lotes,
devoluciones) están fuera del alcance de v1.0.

--- BLOQUEANTES ---------------------------------------------------------------

[BACK-B1] Sin herramienta de migraciones de base de datos (Flyway / Liquibase)
  Archivo: pom.xml (ausente), application.yaml (ddl-auto: validate)
  Descripción: No existe ningún script SQL de creación del schema. No hay
  data.sql, schema.sql ni migraciones versionadas. El schema en producción debe
  crearse manualmente. Con ddl-auto: validate, si el schema no existe o está
  desactualizado, la aplicación NO ARRANCA y no hay mensaje que indique qué
  falta exactamente.
  Sin migraciones no es posible:
    · Reproducir la BD en un nuevo servidor sin documentación extra.
    · Rastrear cambios de schema entre versiones.
    · Hacer rollback de un cambio de schema.
  Acción: generar el script DDL inicial desde el ORM (spring.jpa.generate-ddl:
  true una vez para exportar), versionarlo como V1__init_schema.sql y agregar
  la dependencia de Flyway al pom.xml. Para v1.0, un único script SQL es
  aceptable — no es necesaria la complejidad completa de Flyway si el equipo
  prefiere scripts manuales, pero el script DEBE existir en el repositorio.

[BACK-B2] Sin Dockerfile ni docker-compose.yml
  El despliegue es enteramente manual. No hay forma reproducible de levantar
  el stack (app + PostgreSQL + frontend + red) en un servidor nuevo. Cada
  despliegue depende de documentación externa o memoria del operador.
  Acción: crear Dockerfile multi-stage para el backend, Dockerfile multi-stage
  para el frontend (Node.js build → nginx-alpine), y un docker-compose.yml con
  los 3 servicios: db, backend y frontend.
  Ver §9 (Procedimiento de instalación) para los archivos concretos.

[BACK-B3] Sin perfil de producción (application-prod.yaml)
  application.yaml es el único archivo de configuración. Contiene:
    · spring.jpa.show-sql: true  ← imprime cada SQL en producción
    · JWT secret con valor por defecto hardcodeado si JWT_SECRET no está
      definido en el entorno
    · Contraseña de BD por defecto (david123) si DB_PASSWORD no está definido
  Si se olvida definir alguna variable de entorno en producción, la aplicación
  arranca con valores de desarrollo inseguros sin ningún aviso.
  Acción: crear src/main/resources/application-prod.yaml con:
    spring.jpa.show-sql: false
    spring.jpa.properties.hibernate.format_sql: false
    spring.jpa.hibernate.ddl-auto: validate
    springdoc.api-docs.enabled: false
    springdoc.swagger-ui.enabled: false
    logging.level.root: WARN
    logging.level.com.codigo2enter.almacenes: INFO
  Y activar con SPRING_PROFILES_ACTIVE=prod en el servidor de producción.

--- IMPORTANTES ---------------------------------------------------------------

[BACK-I1] DataInitializer registra la contraseña del admin en texto plano
  Archivo: DataInitializer.java, línea 53
  Código: log.warn(" Usuario: admin | Password: Admin123!")
  En los logs de arranque del servidor queda impresa la contraseña en claro.
  Cualquier persona con acceso a los logs del servidor la ve inmediatamente.
  Acción: eliminar esa línea. El log útil es solo confirmar que el admin fue
  creado. La contraseña NUNCA debe loggearse.

[BACK-I2] Swagger UI accesible sin autenticación en producción
  Archivo: SecurityConfig.java (permitAll en /swagger-ui/**, /v3/api-docs/**)
  En producción, cualquier persona que llegue al servidor puede explorar toda
  la estructura de la API (endpoints, DTOs, parámetros, respuestas posibles)
  sin autenticarse. Esto facilita la enumeración de la API por atacantes.
  Acción: en application-prod.yaml: springdoc.api-docs.enabled: false
  O restringir la ruta a rangos IP de red interna en SecurityConfig.

[BACK-I3] Sin headers de seguridad HTTP
  Archivo: SecurityConfig.java
  El servidor no envía ninguno de los headers de seguridad estándar:
    · Content-Security-Policy
    · Strict-Transport-Security (HSTS)
    · X-Frame-Options: DENY
    · X-Content-Type-Options: nosniff
    · Referrer-Policy: strict-origin-when-cross-origin
  Acción en SecurityConfig.java (método filterChain):
    http.headers(headers -> headers
        .frameOptions(f -> f.deny())
        .xssProtection(Customizer.withDefaults())
        .contentTypeOptions(Customizer.withDefaults())
        .httpStrictTransportSecurity(hsts -> hsts
            .includeSubDomains(true)
            .maxAgeInSeconds(31536000))
    );
  El CSP completo configurarlo en nginx (más flexible que en Spring).

[BACK-I4] DataIntegrityViolationException no manejada en GlobalExceptionHandler
  Archivo: GlobalExceptionHandler.java
  Si una constraint de BD se viola por race condition (dos requests simultáneos
  crean el mismo username/RFC), PostgreSQL lanza DataIntegrityViolationException
  que llega al catch genérico de RuntimeException y retorna HTTP 500 exponiendo
  el nombre interno de la constraint (ej: "uq_supplier_rfc").
  Acción: agregar handler específico:
    @ExceptionHandler(DataIntegrityViolationException.class)
    public ResponseEntity<ApiError> handleDataIntegrity(...) {
        // Extraer constraint del mensaje para dar mensaje amigable
        // Retornar 409 con mensaje genérico
    }

[BACK-I5] ~15 new RuntimeException() en servicios que deberían ser
  ResourceNotFoundException
  Archivos: PurchaseOrderServiceImpl, SaleOrderServiceImpl,
  ReservationServiceImpl (~15 ocurrencias total)
  Ejemplo: "Orden con id X no encontrada" lanzada como RuntimeException →
  GlobalExceptionHandler lo captura como HTTP 500, no como 404.
  Esto significa que un GET a una orden inexistente devuelve 500 en lugar de 404,
  contamina los logs de producción (los errores 500 parecen bugs del sistema)
  y dificulta el monitoreo.
  Acción: buscar con grep "new RuntimeException" en los servicios y reemplazar
  por ResourceNotFoundException o BusinessRuleException según corresponda.

[BACK-I6] Login devuelve HTTP 500 para credenciales incorrectas (no 401)
  El backend lanza BadCredentialsException que sí es capturada por
  GlobalExceptionHandler y retorna 401, PERO si hay algún otro error en el
  flujo de autenticación, puede retornar 500. El frontend ya lo maneja leyendo
  err.error?.message independientemente del status code. Sin embargo, en
  producción, cada login fallido por contraseña incorrecta genera un 500 en
  los logs, lo que hace imposible distinguir errores reales del servidor de
  intentos de login fallidos con alertas basadas en tasa de 5xx.
  Acción: verificar que AuthController siempre retorna 401 para credenciales
  incorrectas (nunca 500). Si hay un caso que llega a 500, corregirlo.

[BACK-I7] Sin configuración de HikariCP
  application.yaml no tiene spring.datasource.hikari.*. Se usan los defaults:
  pool de 10 conexiones, timeout 30 segundos. Para producción con carga normal
  de un almacén, 10 conexiones puede ser insuficiente durante picos (generación
  de reportes analíticos que ejecutan múltiples queries simultáneas).
  Acción en application-prod.yaml:
    spring.datasource.hikari.maximum-pool-size: 20
    spring.datasource.hikari.minimum-idle: 5
    spring.datasource.hikari.connection-timeout: 30000
    spring.datasource.hikari.idle-timeout: 600000
    spring.datasource.hikari.max-lifetime: 1800000

[BACK-I8] Sin Spring Boot Actuator
  pom.xml no incluye spring-boot-starter-actuator. Sin él no hay:
    · /actuator/health para verificar que el sistema está vivo
    · Liveness/readiness probes para load balancers
    · Métricas de JVM y pool de conexiones
    · Smoke tests automatizados post-deploy
  Acción: agregar al pom.xml y configurar en application-prod.yaml:
    management.endpoints.web.exposure.include: health,info
    management.endpoint.health.show-details: never
    management.server.port: 8081  # Puerto separado, no expuesto al público

[BACK-I9] Sin logback-spring.xml (logging estructurado para producción)
  Spring Boot usa su configuración por defecto: solo consola, nivel INFO, sin
  rotación, sin formato JSON. En producción esto significa:
    · Los logs se pierden al reiniciar el proceso
    · No hay integración posible con sistemas de monitoreo (ELK, Grafana Loki)
    · No hay rotación automática (el log crece indefinidamente en disco)
  Acción: crear src/main/resources/logback-spring.xml con:
    · Perfil dev: consola con colores
    · Perfil prod: archivo rotativo diario + retención 30 días + nivel WARN
      para librerías externas, INFO para com.codigo2enter.almacenes

[BACK-I10] Sin logging de operaciones críticas de negocio
  Ningún servicio tiene @Slf4j + registro de las operaciones de negocio
  importantes. En producción no hay trazabilidad de:
    · Login exitoso/fallido con username e IP de origen
    · Cambios de contraseña
    · Creación/desactivación de usuarios
    · Transiciones de estado de órdenes (quién aprobó OC-123 y cuándo)
    · Movimientos de stock (quién registró qué movimiento)
  Los campos de auditoría en la BD (createdByUsername, approvedByUsername, etc.)
  cubren la trazabilidad de datos, pero no hay registro en logs de los intentos
  fallidos ni de las acciones exitosas en tiempo real.
  Acción: agregar @Slf4j en los ServiceImpl críticos y log.info() en las
  operaciones de negocio con username, id de entidad y acción.

[BACK-M3] Endpoints auxiliares sin paginación — sin impacto real en v1.0

    Ambos aceptan ?page=0&size=20. Es el endpoint que alimenta las tabs

  · Endpoints auxiliares por filtro → NO paginados (retornan List<>)

    GET /purchases/orders/supplier/{supplierId}/status/{status}
    GET /purchases/orders/product/{productId}
    GET /sales/orders/client/{clientId}

  Riesgo real en v1.0: NULO. No hay peticiones sin paginar en producción.
  Acción v1.1: si se conectan estos endpoints a alguna vista de historial
  (órdenes de un proveedor, historial de un cliente), paginarlos en ese

[BACK-M4] Sin @Cacheable en catálogos de lectura frecuente

8.1 PLAN DE ATENCIÓN DE BRECHAS — PROTOCOLO DE IMPLEMENTACIÓN
================================================================================

PRINCIPIOS GENERALES
--------------------
  · Una sola brecha a la vez — no se inicia la siguiente hasta que la actual
    tiene criterios de éxito cumplidos y tests en verde.
  · Cada brecha va en su propia rama: fix/<ID> (ej. fix/BACK-B1).
  · El merge a develop usa --no-ff (registro de la rama en el historial).
  · Si una brecha falla sus criterios de éxito, se revierte antes de continuar.
  · El número de orden del §8 define la secuencia de ejecución.

PROCESO ESTÁNDAR POR BRECHA
----------------------------
  1. git checkout develop && git pull origin develop
  2. git checkout -b fix/<ID>
  3. Implementar los cambios (solo los descritos en esta sección)
  4. Ejecutar los tests específicos de la brecha
  5. Ejecutar la suite completa del componente afectado
  6. Ejecutar la suite completa del sistema (regresión)
  7. Validar en browser con los 4 roles (si la brecha afecta UI/API)
  8. Confirmar criterios de éxito
  9. git checkout develop && git merge --no-ff fix/<ID>
  10. Marcar la brecha como RESUELTA en este documento (columna Estado)

ESTRATEGIA DE ROLLBACK
-----------------------
  · Si la brecha está en su rama (paso 2-8): git checkout develop && git branch -D fix/<ID>
  · Si ya se mergeó y se detecta el problema después:
      git revert -m 1 <hash-del-merge-commit>
      git push origin develop
  · Para brechas de configuración (application.yaml, Dockerfiles): cada cambio
    incluye el valor original como comentario para restauración manual inmediata.
  · Para brechas de BD (BACK-B1): el script DDL solo se aplica una vez en el
    primer despliegue; en desarrollo el checkpoint es pg_dump antes de aplicarlo.

[36] BACK-M1 — FetchType.LAZY en User.roles             Estado: ✅ RESUELTA
──────────────────────────────────────────────────────────────────────────────
Cambios necesarios:

  · Línea 50: cambiar @ManyToMany(fetch = FetchType.EAGER) a FetchType.LAZY.
  · CRÍTICO: con LAZY, si Spring Security intenta acceder a user.getRoles()
    fuera de una sesión Hibernate activa, lanza LazyInitializationException.
  · Spring Security necesita los roles al cargar el UserDetails en
    JwtAuthenticationFilter — ese acceso debe ocurrir DENTRO de una transacción.
  · Verificar que JwtAuthenticationFilter o el método que carga el UserDetails
    tiene @Transactional o usa Hibernate.initialize().

Archivos afectados:
  · User.java (1 línea)
  · Posiblemente UserServiceImpl o JwtAuthenticationFilter (si necesitan @Transactional)

Daño colateral posible:
  · LazyInitializationException en TODOS los requests autenticados si los roles
    se acceden fuera de una sesión Hibernate activa — esto rompe toda la seguridad.
  · Este es el riesgo más alto de todos los cambios del backend.

Cómo evitarlo:
  · ANTES de cambiar a LAZY, agregar @Transactional al método que carga el
    UserDetails (UserServiceImpl.loadUserByUsername o equivalente).
  · Alternativa más segura: usar JOIN FETCH en la query del repositorio:
      @Query("SELECT u FROM User u JOIN FETCH u.roles WHERE u.username = :username")
      Optional<User> findByUsernameWithRoles(@Param("username") String username);
  · Esta alternativa (JOIN FETCH) es preferida porque no requiere @Transactional
    en el filtro de seguridad.

10. PROCEDIMIENTO DE BACKUP Y RECUPERACIÓN ANTE DESASTRES
================================================================================

POLÍTICA DE BACKUP
------------------
  · Tipo:       Lógico (pg_dump — SQL portable)
  · Frecuencia: Diario, a las 02:00 AM (horario de menor actividad)
  · Retención:  Local: últimos 30 días de backups diarios (RETENTION_DAYS=30).
                Offsite (opcional): archivado mensual hasta 12 meses.
  · Verificación: Restaurar en entorno de prueba el primer lunes de cada mes
  · Almacenamiento: Local + copia remota opcional (S3 o SFTP)
                (el INSTRUCTIVO del backend es la referencia operativa del script)

SCRIPT DE BACKUP (/opt/almacenes/scripts/backup.sh)
----------------------------------------------------
  #!/bin/bash
  set -euo pipefail

  BACKUP_DIR="/var/backups/almacenes"
  TIMESTAMP=$(date +%Y%m%d_%H%M%S)
  FILENAME="almacenes_db_${TIMESTAMP}.sql.gz"
  RETENTION_DAYS=30

  # Crear directorio si no existe
  mkdir -p "${BACKUP_DIR}"

  # Generar backup comprimido
  PGPASSWORD="${POSTGRES_PASSWORD}" pg_dump \
      -h localhost -U almacenes_user -d almacenes_db \
      --no-owner --no-acl \
      | gzip > "${BACKUP_DIR}/${FILENAME}"

  # Verificar que el archivo no está vacío
  if [ ! -s "${BACKUP_DIR}/${FILENAME}" ]; then
      echo "ERROR: El backup está vacío. Verificar conexión a BD."
      exit 1
  fi

  echo "Backup completado: ${BACKUP_DIR}/${FILENAME} ($(du -h "${BACKUP_DIR}/${FILENAME}" | cut -f1))"

  # Eliminar backups más antiguos que la retención configurada
  find "${BACKUP_DIR}" -name "almacenes_db_*.sql.gz" -mtime +${RETENTION_DAYS} -delete

  # (Opcional) Copiar a almacenamiento remoto
  # aws s3 cp "${BACKUP_DIR}/${FILENAME}" s3://bucket-backups/almacenes/

CONFIGURAR CRON (/etc/cron.d/almacenes-backup)
----------------------------------------------
  # Backup diario a las 02:00 AM
  0 2 * * * root /bin/bash /opt/almacenes/scripts/backup.sh >> /var/log/almacenes-backup.log 2>&1

PROCEDIMIENTO DE RESTAURACIÓN
------------------------------
  1. Identificar el backup a restaurar:
     ls -lh /var/backups/almacenes/

  2. Detener el backend para evitar escrituras durante la restauración:
     sudo systemctl stop almacenes-backend

  3. Crear una BD vacía nueva (o limpiar la existente):
     sudo -u postgres psql -c "DROP DATABASE IF EXISTS almacenes_db_restore;"
     sudo -u postgres psql -c "CREATE DATABASE almacenes_db_restore OWNER almacenes_user;"
     sudo -u postgres psql -c "\c almacenes_db_restore; CREATE EXTENSION unaccent;"
     sudo -u postgres psql -c "\c almacenes_db_restore; CREATE OR REPLACE FUNCTION f_unaccent..."

  4. Restaurar el backup:
     gunzip -c /var/backups/almacenes/almacenes_db_20260617_020000.sql.gz \
         | PGPASSWORD="${POSTGRES_PASSWORD}" psql -h localhost -U almacenes_user almacenes_db_restore

  5. Verificar la integridad básica:
     PGPASSWORD="${POSTGRES_PASSWORD}" psql -h localhost -U almacenes_user almacenes_db_restore \
         -c "SELECT COUNT(*) FROM products; SELECT COUNT(*) FROM sale_orders;"

  6. Si la restauración es correcta, hacer swap de la BD activa:
     sudo -u postgres psql -c "ALTER DATABASE almacenes_db RENAME TO almacenes_db_old;"
     sudo -u postgres psql -c "ALTER DATABASE almacenes_db_restore RENAME TO almacenes_db;"

  7. Reiniciar el backend:
     sudo systemctl start almacenes-backend

  8. Ejecutar el smoke test (ver §11) para verificar el sistema.

  9. Una vez verificado, eliminar la BD vieja:
     sudo -u postgres psql -c "DROP DATABASE almacenes_db_old;"

TIEMPO OBJETIVO DE RECUPERACIÓN (RTO)
--------------------------------------
  · RPO (Punto de recuperación): máximo 24 horas (con backup diario)
  · RTO (Tiempo de recuperación): máximo 2 horas para restauración completa
  · Para reducir RPO a horas: implementar backup cada 4 horas (cambiar cron)
  · Para reducir RPO a minutos: implementar replicación de streaming de PostgreSQL
    (requiere servidor secundario — fuera del alcance de v1.0)

12. ALCANCE FUNCIONAL DECLARADO v1.0 Y RESTRICCIONES CONOCIDAS
================================================================================

ALCANCE v1.0
  · Un solo almacén (no multi-warehouse)
  · Sin gestión de ubicaciones físicas (sin zonas/pasillos/estantes)
  · Sin trazabilidad de lotes ni números de serie
  · Sin estrategia de picking automatizada (FIFO/FEFO)
  · Sin módulo de devoluciones (workaround: movimiento manual de stock)
  · Sin alertas automáticas push (workaround: revisar reporte de stock bajo
    manualmente al inicio del turno)
  · Sin inventario físico asistido por sistema (workaround: usar movimientos
    de ajuste con motivo "AJUSTE INVENTARIO FÍSICO YYYY-MM-DD")
  · Sin importación masiva de datos desde CSV/Excel (carga manual o via API)
  · JWT almacenado en localStorage (no HttpOnly cookie) — riesgo XSS mitigado
    con CSP y dado que la app no renderiza contenido HTML externo no controlado
  · Sin manejo de sesiones concurrentes — JWT válido por 2 horas sin posibilidad
    de invalidación anticipada (consecuencia de arquitectura stateless)

PROCEDIMIENTO OPERATIVO WORKAROUNDS v1.0
  · Devolución de cliente: registrar un movimiento IN con motivo
    "DEVOLUCION CLIENTE - OV-NNN - MOTIVO"
  · Devolución a proveedor: registrar un movimiento OUT con motivo
    "DEVOLUCION PROVEEDOR - OC-NNN - MOTIVO"
  · Inventario físico anual: comparar Kardex del sistema con conteo físico,
    registrar movimiento de ajuste (IN si el sistema tiene menos, OUT si tiene
    más) con motivo "AJUSTE INVENTARIO FÍSICO YYYY-MM-DD"
  · Alertas de stock bajo: WAREHOUSEMAN debe revisar Reportes → Operativos →
    Stock Bajo cada mañana al inicio de turno

13. DEUDA TÉCNICA DOCUMENTADA PARA v1.1+
================================================================================

Esta sección documenta las decisiones de diseño y funcionalidades que se han
evaluado conscientemente y se difieren para versiones posteriores. No son bugs
ni omisiones — son decisiones de alcance.

MÓDULOS FUNCIONALES EN ROADMAP
  · Módulo Returns: devoluciones de clientes y a proveedores. Prioridad alta
    si el cliente tiene volumen de devoluciones > 5% de transacciones.
  · Módulo Notifications: alertas automáticas por email/websocket para stock
    bajo y órdenes vencidas.
  · Módulo Locations: gestión de ubicaciones físicas (zonas, estantes, bins).
    Prerrequisito para FIFO/FEFO automatizado.
  · Módulo Shipping: soporte para despachos con número de guía/transportista.

DECISIONES TÉCNICAS ACEPTADAS
  · Sin Flyway para v1.0: el schema se mantiene con script SQL manual.
    Prioridad alta para v1.1 cuando aparezca el primer cambio de schema.
  · JWT en localStorage: alternativa es HttpOnly cookie + endpoint de refresh,
    requiere cambios en frontend Y backend. Evaluar en v1.1 según el nivel
    de exposición de la aplicación a internet.
  · Tests E2E automatizados: YA IMPLEMENTADOS (OPS-B6 resuelto — 6 specs en
    frontend e2e/). La cobertura unitaria es excelente (88.94% statements).
    Deuda v1.1: ampliar la batería E2E si se agregan flujos nuevos.
  · Sin Rate Limiting global (solo en login): agregar en v1.1 si hay exposición
    a internet con Spring Security Rate Limiter o API Gateway.
  · Tabla de auditoría general ausente: la trazabilidad actual por campos en
    las entidades es suficiente para v1.0. Una tabla audit_log completa
    (toda entidad, todos los campos) es v1.1+.

PRODUCTION-READINESS / DEVOPS (v1.1+)
  Evaluación contra estándares de la industria (SRE Production Readiness Review,
  DORA, OWASP ASVS, 12-Factor, Well-Architected). Ninguno es bloqueante para la
  escala declarada (un almacén, <100 usuarios), pero son las mejoras que llevarían
  el sistema a un "production-ready" pleno. Se difieren conscientemente a v1.1+.

  Prioridad alta (bajo esfuerzo, alto valor):
    · CI/CD (GitHub Actions): pipeline que corra build + tests + lint en cada PR/push
      en ambos repos. Hoy el gatekeeper es manual (DORA — automatización).
    · Supply chain / seguridad de dependencias: Dependabot + npm audit / OWASP
      Dependency-Check + escaneo de secretos (gitleaks) (OWASP ASVS).
    · Drills de go-live ejecutados en staging antes de prod: ensayo del despliegue
      completo, prueba de restauración de backup y prueba de rollback (SRE).

  Prioridad media (observabilidad y resiliencia):
    · Observabilidad: exponer métricas de Actuator (metrics/prometheus, hoy solo
      health,info) + alertas mínimas (caída, tasa de 5xx, disco) (SRE).
    · Monitoreo de uptime externo (synthetic check a /actuator/health o login).
    · Reliability: límites de cpu/mem en docker-compose + server.shutdown: graceful.
    · Performance: prueba de carga básica (k6/JMeter) para validar la concurrencia
      esperada (los reportes analíticos son query-heavy).

  Consideración no técnica:
    · Protección de datos personales (clientes/proveedores — PII): revisar
      obligaciones según jurisdicción (p. ej. LFPDPPP en México). Decisión de negocio.
