1. CONTEXTO Y JUSTIFICACIÓN
--------------------------------------------------------------------------------

El Módulo 3 — Compras gestiona el ciclo de vida completo de las órdenes de
compra: desde la creación hasta la recepción física de mercancía en el almacén.
Es el punto de integración crítico entre proveedores e inventario.

El backend está completamente implementado y probado (0 fallos). El frontend
consume 13 endpoints de órdenes + 4 endpoints de proveedores.

Audiencias:
  - ADMIN / MANAGER : acceso completo (CRUD proveedores, crear/aprobar/recibir/cancelar órdenes)
  - WAREHOUSEMAN    : solo recepción de órdenes aprobadas (PATCH /receive)
  - SALES           : sin acceso al módulo

La recepción de una orden APPROVED es la única acción que incrementa el
inventario físico de forma automática (registerStockMovement IN por detalle).

  POST   /purchases/suppliers              → 201  SupplierDTO
  GET    /purchases/suppliers/active       → 200  PageResponse<SupplierDTO>

  RBAC backend:
    GET  → ADMIN, MANAGER, WAREHOUSEMAN  (SALES no tiene acceso a purchases)
    POST, PUT, DELETE → ADMIN, MANAGER

─── ÓRDENES DE COMPRA ───────────────────────────────────────────────────────

  POST   /purchases/orders                          → 201  PurchaseOrderResponseDTO
  GET    /purchases/orders/{id}                     → 200  PurchaseOrderResponseDTO
  GET    /purchases/orders/status/{status}          → 200  PageResponse<PurchaseOrderResponseDTO>
  GET    /purchases/orders/supplier/{supplierId}    → 200  List<PurchaseOrderResponseDTO>  (no paginada)
  GET    /purchases/orders/supplier/{sid}/status/{s}→ 200  List<PurchaseOrderResponseDTO>  (no paginada)
  GET    /purchases/orders/product/{productId}      → 200  List<PurchaseOrderResponseDTO>  (no paginada)
  PUT    /purchases/orders/{id}                     → 200  PurchaseOrderResponseDTO
  PATCH  /purchases/orders/{id}/approve             → 200  PurchaseOrderResponseDTO  (sin body)
  PATCH  /purchases/orders/{id}/receive             → 200  PurchaseOrderResponseDTO  (sin body)
  PATCH  /purchases/orders/{id}/cancel              → 200  PurchaseOrderResponseDTO  (sin body)
  POST   /purchases/orders/{id}/details             → 201  PurchaseOrderResponseDTO  (orden completa)
  PUT    /purchases/orders/{id}/details/{detailId}  → 200  PurchaseOrderResponseDTO  (orden completa)
  DELETE /purchases/orders/{id}/details/{detailId}  → 204  void

  RBAC backend:
    GET      → ADMIN, MANAGER, WAREHOUSEMAN
    PATCH /receive → ADMIN, MANAGER, WAREHOUSEMAN
    POST, PUT, PATCH (aprobar/cancelar), DELETE → ADMIN, MANAGER

─── DTOs VERIFICADOS (nombres exactos de campos) ────────────────────────────

  SupplierDTO:
    id, rfc, companyName, contactName, phone, email, address, active
    createdAt, createdById, createdByUsername
    updatedAt, updatedById, updatedByUsername

  PurchaseOrderRequestDTO (body de creación):
    supplierId (Long, requerido)
    notes (String, opcional, max 500)
    details (List<PurchaseOrderDetailRequestDTO>, @NotEmpty)
      └─ productId (Long, requerido)
         quantity (int, min 1, requerido)
         unitPrice (BigDecimal, min 0.01, requerido)

  PurchaseOrderResponseDTO:
    id, orderNumber, status, notes, totalAmount
    supplierId, supplierName
    createdById, createdByUsername, createdAt, updatedAt
    approvedAt, approvedById, approvedByUsername
    receivedAt, receivedById, receivedByUsername
    cancelledAt, cancelledById, cancelledByUsername
    details (List<PurchaseOrderDetailResponseDTO>)
      └─ id, quantity, unitPrice, subtotal
         productId, productSku, productName

  PurchaseOrderUpdateRequestDTO (body de PUT /{id}):
    supplierId (Long, requerido)
    notes (String, opcional)

  PurchaseOrderDetailUpdateRequestDTO (body de PUT /{id}/details/{detailId}):
    quantity (int, min 1, requerido)
    unitPrice (BigDecimal, min 0.01, requerido)
    — productId NO se incluye (cambio de producto = delete + add)

─── CAMPOS DE SOLO LECTURA (el backend los ignora o los genera) ─────────────

  orderNumber   → generado por el servicio (OC-YYYY-NNNN) — nunca editable
  totalAmount   → calculado por el servicio (suma de subtotales) — nunca editable
  subtotal      → calculado por el servicio (quantity × unitPrice) — nunca editable
  status        → solo cambia por PATCH /approve, /receive, /cancel — no editable directo
  createdAt/updatedAt/approvedAt/receivedAt/cancelledAt → auditoría — solo lectura
  createdBy*, approvedBy*, receivedBy*, cancelledBy* → auditoría — solo lectura

─── REGLAS DE NEGOCIO DEL BACKEND (PurchaseOrderServiceImpl) ────────────────

  R1. Edición solo en PENDING: PUT /{id}, POST /{id}/details, PUT y DELETE details
      → si status != PENDING → error 422
  R2. Aprobación requiere al menos un detalle → si details.isEmpty() → error 422
  R3. Recepción solo desde APPROVED → error 422 si status != APPROVED
  R4. Cancelación desde PENDING o APPROVED (no desde RECEIVED) → error 422 si RECEIVED
  R5. Desactivar proveedor bloquea si tiene órdenes PENDING o APPROVED → error 422
  R6. Producto duplicado en misma orden al hacer addDetail → error (validación de servicio)
  R7. unitPrice en el detalle es el precio pactado — puede diferir de Product.price

─── TRAZABILIDAD REGLAS → COMPONENTES UI ────────────────────────────────────

  R1 → botones Editar/Agregar detalle/Eliminar detalle: solo visibles cuando status=PENDING
  R2 → botón Aprobar: deshabilitado preventivamente si details está vacío
  R3 → botón Recibir: solo visible cuando status=APPROVED
  R4 → botón Cancelar: visible en PENDING y APPROVED, oculto en RECEIVED y CANCELLED
  R5 → modal de desactivación de proveedor: advierte si tiene órdenes activas; el backend
       retorna 422 con mensaje — mostrar en snackbar rojo
  R6 → selector de producto en addDetail: deshabilitar productos ya en la orden
  R7 → campo unitPrice del detalle: editable, pre-rellenado con Product.price pero modificable

  D1. Creación de orden → RUTA /purchases/orders/new
      Justificación: la orden necesita formulario de cabecera + gestión de detalles
      (tabla editable con agregar/quitar líneas) en la misma vista. Un MatDialog
      quedaría demasiado comprimido para esa complejidad.

  D2. Tab activa por defecto → PENDIENTES
      Justificación: las órdenes PENDING son las que requieren acción inmediata
      (aprobar, agregar detalles). Es el estado más frecuente en el flujo diario
      del MANAGER, que es el usuario principal del módulo.

  D3. Selector de producto en detalle → MATAUTOCOMPLETE con búsqueda
      El usuario escribe parte del SKU o nombre y filtra en tiempo real.
      Formato de opción: "[SKU] — Nombre del producto (disponible: N unidades)".
      Opciones ya presentes en la orden se muestran deshabilitadas.

  D4. Historial de estado de la orden → TABLA DE 3 FILAS
      Columnas: Estado | Usuario | Fecha.
      Filas: Aprobado / Recibido / Cancelado.
      Si el estado no fue alcanzado, mostrar "—" en usuario y fecha.
      Consistente con el estilo de tabla del proyecto; sin CSS custom adicional.
